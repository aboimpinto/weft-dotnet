# FEAT-001 Framework Capability Comparison

**Snapshot date:** 2026-07-15
**Conclusion status:** Evidence baseline; no universal winner and no reproduced
cross-platform performance result
**Method:** `research-methodology-and-evidence-contract.md`

## Executive Finding

The market is not missing ways to build C# web applications. Blazor is a
mature, supported C# component platform. The narrower gap Weft targets is a
mainstream-quality, HTML-first, feature-package architecture in which C# is the
default language on both sides of the HTTP boundary, JavaScript and a managed
browser runtime are optional per capability, and a project-local JavaScript
dependency installation is not mandatory.

Next.js and Angular provide the broadest integrated application platforms in
this comparison. TanStack Start provides unusually strong typed routing and a
composable collection of headless data/form/table tools, but the developer must
make and maintain more integration choices. Blazor provides the closest
language alternative and the widest supported .NET rendering spectrum, but its
primary unit of composition is a Razor component and interactive render mode.
Current Weft proves only generated, explicitly trusted server feature
registration and durable HTML form handling. Everything beyond that foundation
remains a target hypothesis.

## Coverage Legend

- `E` — `EVIDENCED`: Grade-A capability or limitation.
- `NYI` — `NOT_YET_IMPLEMENTED`: Current Weft repository lacks the capability.
- `NA` — `NOT_APPLICABLE`: outside the product boundary stated in the cell.
- `H` — `EVIDENCED` target hypothesis: documented intent, not current behavior.

Each cell includes its conclusion label (`F` Fact, `I` Inference, `H`
Hypothesis) and evidence IDs. Deployment-mode differences are stated rather
than averaged.

## Matrix Conclusions

1. **Blazor is the decisive comparator, not proof that Weft is unnecessary.**
   It validates C# web demand and covers more rendering modes today. Weft must
   demonstrate a materially simpler HTML-first feature/package contract rather
   than merely reproduce Razor components.
2. **Node platforms win on ecosystem and integrated browser tooling.** Weft
   should interoperate with selected JavaScript components instead of claiming
   that a new .NET ecosystem can replace the npm ecosystem quickly.
3. **The dependency claim must be precise.** NuGet's shared global package
   cache avoids a project-local expanded dependency tree, but it is still an
   expanded cache that grows and requires governance. npm has deterministic
   locks, integrity, audit/provenance and newer script-approval controls. Weft's
   opportunity is a smaller default graph and explicit asset policy—not the
   assertion that npm has no controls.
4. **Security depends more on execution and authority boundaries than on
   language.** Server-rendered C# can reduce client exposure, but every platform
   has secure and insecure modes. Weft can seek a smaller default attack
   surface; it cannot honestly claim categorical security superiority.
5. **No performance winner is established.** The modes are too different and
   equivalent reference implementations do not exist. Phase 6 therefore
   defines methodology only.

## Installation Lifecycle Addendum

The official installation flows reinforce the broader comparison rather than
changing its product conclusion:

- Blazor benefits from being included in the .NET SDK/template ecosystem. .NET
  local tools also supply a strong project-pinning and side-by-side-version
  mechanism that Weft can adopt without inventing a package manager.
- Next.js, Angular, and TanStack provide effective creator commands, but their
  baseline workflows require Node-compatible package installation and a local
  application dependency graph. Angular's documented global CLI path can also
  interact with Windows PowerShell execution policy or Unix global-package
  permissions.
- Tauri's creator is unusually flexible and detects missing dependencies, but
  the chosen target still brings Rust, operating-system build dependencies,
  and—for mobile—Android or Apple toolchains. Its short remote-script paths do
  not eliminate that prerequisite graph.
- Current Weft has no distributable CLI, template package, installer, `doctor`,
  update/migration flow, offline restore contract, or conservative removal
  mechanism. These remain target requirements, not current capability.

The product proposal is specified in
`installation-and-bootstrap-strategy.md`: a verified versioned release archive
is the experimental bootstrap, a pinned repository-local .NET tool is the
mature team/CI default, global installation is optional convenience, and the
pure HTML profile requires only the .NET SDK. `wget` is a supported Unix
transport, not a hidden package manager and not a reason to execute an
unverified mutable script.
